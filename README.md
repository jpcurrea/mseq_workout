# Workout Routine App

A personal workout tracking app with a Flutter frontend and FastAPI backend. Tracks a rotating workout schedule generated by m-sequences, lets you log scores, and view history for any past date.

---

## Architecture

| Layer | Technology |
|---|---|
| Frontend | Flutter (targets web, Android, iOS, Windows) |
| Backend | FastAPI (Python) |
| Database | SQLite via SQLAlchemy |
| Auth | Google OAuth 2.0 + JWT |
| Deployment | Netlify (frontend) + Render (backend) |

---

## Features

### Workout Schedule
- **Today's workout** — the home screen shows the workouts scheduled for today. If today has no scheduled workout (rest day), it automatically falls back to the most recent past scheduled date so you always know what you last did.
- **Date picker** — a calendar icon on the home screen lets you jump to any past date to view and edit scores for that day. The same backwards-lookup applies: selecting a rest day shows the last workout day before it.
- **Score entry** — each workout row has an inline score field. Hit "Submit All Scores" to save them in one shot.

### Workout Management
- Add, edit, and delete workouts (name, goal, units, home vs park)
- Each workout has a goal and units (e.g. `50 reps`, `5 km`)
- Home vs park distinction shown with colour-coded icons

### Routine Generation
- Generates a new workout schedule using **m-sequences** (maximum-length sequences), a pseudorandom binary pattern that guarantees each workout appears roughly the same number of times with no two consecutive repeats in a row
- Schedules workouts every other day
- Preserves existing score history when regenerating

### History
- History screen shows a calendar-style log of all past sessions
- Past scores pre-fill in the input fields so you can correct an entry

### Export
- `GET /export` endpoint returns the full schedule as a CSV file

## Running Locally

### Backend

```cmd
cd mobile_app\backend
..\.venv\Scripts\pip install -r requirements.txt
..\.venv\Scripts\python -m uvicorn main:app --reload --port 8000
```

Requires a `.env` file in the `backend/` folder:

```
SECRET_KEY=<generate with: python -c "import secrets; print(secrets.token_urlsafe(32))">
GOOGLE_CLIENT_ID=<from Google Cloud Console>
GOOGLE_CLIENT_SECRET=<from Google Cloud Console>
FRONTEND_URL=http://localhost:8080
```

### Flutter Frontend

```cmd
cd mobile_app\flutter_app
flutter pub get
flutter run -d chrome --web-port 8080
```

The web port must match `FRONTEND_URL` in `.env` for OAuth redirects to work.

## API Endpoints

| Method | Path | Description |
|---|---|---|
| GET | `/workouts` | List all workouts |
| POST | `/workouts` | Create a workout |
| PUT | `/workouts/{name}` | Update a workout |
| DELETE | `/workouts/{name}` | Delete a workout |
| GET | `/today` | Today's schedule (or last scheduled day) |
| GET | `/schedule/{date}` | Schedule for a specific date (or last scheduled day before it) |
| POST | `/update-score` | Save a score for a workout on a date |
| POST | `/generate-routine` | Generate a new m-sequence schedule |
| GET | `/export` | Download full schedule as CSV |
| GET | `/auth/google/login` | Start Google OAuth flow |
| GET | `/auth/google/callback` | OAuth callback (handled by backend) |

## Security

### What is in place

**Google OAuth 2.0 — you never give the app your password**
The app never sees or stores your Google password. Clicking "Sign in with Google" redirects you to Google's own login page. Google authenticates you and sends the backend a token confirming who you are. The app only ever receives your name and email address from Google. This is how most major apps (Spotify, Airbnb, etc.) handle login.

**JWT tokens (JSON Web Tokens)**
After Google confirms your identity, the backend issues its own short-lived token (24-hour expiry) that the Flutter app stores and sends with every API request. The backend verifies this token cryptographically before honouring any request. Tokens are signed with a secret key — they cannot be forged.

**Secure token storage**
The JWT is stored using `flutter_secure_storage`, which uses the OS-level keychain (iOS Keychain, Android Keystore, Windows Credential Manager) rather than plain app storage. This means the token is encrypted at rest and not accessible to other apps, even on a rooted device.

**CORS locked to your frontend domain**
The backend only accepts browser-based requests from the exact `FRONTEND_URL` defined in the environment. A malicious website cannot make cross-origin requests to your API.

**Rate limiting**
Write endpoints (score updates, workout CRUD, routine generation) are rate-limited per IP using `slowapi`. This prevents automated scripts from hammering the API.

**All data is user-scoped**
Every database query filters by `user_id`. One authenticated user cannot read or modify another user's data.

**Session cookie hardening**
The OAuth session cookie uses `SameSite=Lax`, which blocks it from being sent in most cross-site request scenarios.

### Remaining limitations to be aware of

**HTTP in local development**
Locally the app runs over plain HTTP, which means the JWT token is transmitted unencrypted on your local network. In production on Render/Netlify everything goes over HTTPS, so this is not a concern for deployed use.

**No token revocation**
If someone stole your JWT (unlikely given secure storage, but possible if a device is compromised), the backend has no way to invalidate it before its 24-hour expiry. To sign out you can clear the token from the app, but a copy of the token obtained by an attacker before that would still work until it expires. A proper revocation list would fix this but adds complexity; for a personal fitness app this risk is low.

**SQLite on a shared server**
SQLite stores all data in a single file on the server's disk. On Render's free tier the disk is ephemeral — data resets on each deploy. Using Render's persistent disk or switching to PostgreSQL would fix this for production.

**No HTTPS enforcement on the session cookie**
The session cookie has `https_only=False` so that local development works. In production this should be `True` — otherwise the cookie could theoretically be sent over HTTP if the redirect chain ever produces an HTTP request. In practice, Render and Netlify enforce HTTPS redirects, so this is low risk.

**`default_user` fallback**
Unauthenticated requests fall back to `default_user` rather than being rejected. This was useful during development but means someone who knows your API URL can read `default_user`'s data without a token. Once all data has been transferred to your Google account and `default_user` has no data, this is essentially harmless.

### What you do not need to worry about

- **Your Google password** — the app never touches it. OAuth is explicitly designed to prevent this.
- **SQL injection** — all database queries go through SQLAlchemy's ORM with parameterised queries.
- **Token forgery** — JWTs are cryptographically signed; without the `SECRET_KEY` an attacker cannot create a valid token.
- **Other users reading your data** — every query is filtered by your `user_id`.

## Deployment

### Backend (Render)

Set these environment variables in the Render dashboard:

| Variable | Value |
|---|---|
| `SECRET_KEY` | Random 32-byte hex string |
| `GOOGLE_CLIENT_ID` | From Google Cloud Console |
| `GOOGLE_CLIENT_SECRET` | From Google Cloud Console |
| `FRONTEND_URL` | Your Netlify URL (e.g. `https://yourapp.netlify.app`) |

### Frontend (Netlify)

Build command: `flutter build web --release`  
Publish directory: `flutter_app/build/web`

Update `baseUrl` in `flutter_app/lib/services/api_service.dart` to your Render URL before building.
